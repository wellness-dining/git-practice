# git-practice｜git操作練習用リポジトリ

このリポジトリは **社内git導入プロジェクトの練習用** です。
本番リポジトリでの作業に慣れるため、まずはここで安全に git 操作を試してみましょう。

---

## 🎯 このリポジトリの目的

- 本番リポジトリを誤操作する前に、安全な環境で git の流れを体験する
- ブランチ作成 → 編集 → コミット → プッシュ → プルリクエスト（PR）→ マージ の一連の流れを身につける
- コンフリクト（競合）の発生と解決、PRのレビュー、ミスの復旧まで体験する

**⚠️ 本番のソースコード・顧客情報・パスワード等は絶対にコミットしないでください。**

## 🏃 練習課題（順番にチャレンジ）

以下7つの課題を順番に進めると、実運用で必要なgit操作がひと通り身につきます。
課題1〜3は全員必須、課題4は責任者向け、課題5〜7は全員推奨です。

| # | 課題 | 対象 | 所要時間 | 学べること |
|---|---|---|---|---|
| 1 | クローン | 全員 | 5分 | リポジトリの取得 |
| 2 | キャッチコピー変更 | 全員 | 15分 | 基本フロー（ブランチ→PR→マージ） |
| 3 | スタッフ一覧に自分を追加 | 全員 | 15分 | **コンフリクト解決** |
| 4 | プルリクエストをレビュー | 責任者 | 10分 | PRレビュー・Approve/Request changes |
| 5 | pullを忘れて作業した時の対処 | 全員 | 15分 | **実運用で頻発するトラブル対処** |
| 6 | 間違えた時の取り消し方 | 全員 | 15分 | **安全な復旧方法** |
| 7 | 色を変える（レビュー却下練習） | 全員 | 10分 | CSSの編集＆レビュー却下体験 |

### 課題1：リポジトリをクローンする（全員）

GitHub Desktop を使って、このリポジトリを自分のPCにコピーします。

1. GitHub Desktop を起動
2. 「File」→「Clone repository」
3. 一覧から `wellness-dining/git-practice` を選択
4. 保存先を確認して「Clone」

**ゴール**：自分のPCに `git-practice` フォルダが作られる。

---

### 課題2：はじめてのブランチ＆プルリクエスト

トップページ（index.html）のキャッチコピーを編集してみます。

1. GitHub Desktop で **新しいブランチを作成**
   - ブランチ名：`feature/キャッチコピー変更-あなたの名前`
   - 例：`feature/catchphrase-yamanoi`
2. `index.html` をテキストエディタで開く
3. `<h1>健康的な食事を、<br>毎日の食卓に。</h1>` の文章を自由に変更
4. GitHub Desktop に戻り、**コミット**（メッセージ例：「キャッチコピーを変更」）
5. **「Push origin」** でGitHubにプッシュ
6. ブラウザでGitHubを開き、**Pull Request（PR）を作成**
7. 責任者にレビューしてもらい、マージ

**ゴール**：自分の変更が `main` ブランチに反映され、GitHub Pagesのサイトに表示される。

---

### 課題3：スタッフ紹介ページに自分を追加（コンフリクト練習）

`staff.html` のスタッフ一覧に、自分の情報を追加します。
**全員が同じファイルの同じ場所を編集するため、コンフリクトが発生する可能性大**。

1. 最新の `main` を pull（GitHub Desktopで「Fetch origin」→「Pull origin」）
2. 新しいブランチ `feature/staff-あなたの名前` を作成
3. `staff.html` の以下の部分を探して、自分の情報に書き換え：
   ```html
   <tr>
     <td>（ここにあなたの名前）</td>
     <td><span class="staff-role">（役職）</span></td>
     <td>（担当業務）</td>
     <td>（ひとことコメント）</td>
   </tr>
   ```
4. コミット → プッシュ → PR作成 → マージ

**コンフリクトが起きたら**：
- GitHubのPR画面で「This branch has conflicts」と表示される
- 責任者と相談して、どちらの変更を残すか決める
- エディタ上で `<<<<<<<` `=======` `>>>>>>>` マークを手動で編集して解決

**ゴール**：公開ページで自分の名前が見えるようになる。

---

### 課題4：プルリクエストをレビューする（責任者向け）

責任者が毎日行う作業の練習です。スタッフが作ったPRをレビューして、承認 or 修正依頼をします。

**役割分担**：
- **責任者役**：レビューする側（この課題の主役）
- **スタッフ役**：課題2〜3で作った未マージのPRを使う

**レビュー手順**：

1. GitHub上でリポジトリを開く
2. 上部メニューの「**Pull requests**」タブをクリック
3. 未マージのPRを選択（例：課題3で誰かが作ったスタッフ追加PR）
4. 「**Files changed**」タブをクリック → 変更内容を確認
5. コード行の左端「**＋**」アイコンをクリックしてコメントを付ける
   - 例：「スペースが2つ入っています」「OK」「ここを○○に変更してください」
6. 右上の「**Review changes**」ボタンをクリック → 以下から選択：

| 選択肢 | 意味 | 使いどころ |
|---|---|---|
| **Comment** | 単純なコメント | 軽いフィードバック。マージは止めない |
| **Approve** | 承認 | 問題ないのでマージしてOK |
| **Request changes** | 修正依頼 | 直してから再提出してほしい（マージがブロックされる） |

7. 「**Submit review**」ボタンをクリック
8. Approveした場合、「**Merge pull request**」→「Confirm merge」でマージ完了

**ゴール**：
- Approve／Request changes／Comment の使い分けができる
- マージボタンの位置と操作を体得する
- コメントの書き方に慣れる

**ポイント**：
- **Approve だけでは自動マージされません**。最後に手動で「Merge pull request」が必要
- **Request changes** にすると、修正されるまでマージボタンがロックされる（強制力あり）
- コメントは改善提案として相手に届くので、**丁寧な言葉遣い**を心がけましょう

### 課題5：pullを忘れて作業した時の対処（実運用で頻発）

実運用で一番起きやすいトラブル「**他の人の変更を取り込まずに作業してしまった**」の体験です。

**このトラブルが起きる流れ**：

1. Aさんが変更をmainにマージ
2. Bさんは自分のPCで作業開始（でもAさんの変更を取り込んでいない）
3. Bさんがプッシュしようとすると「このブランチはmainより遅れています」と表示される

**練習手順**：

1. **意図的に古い状態を作る**：
   - GitHub Desktopで「Fetch origin」を**しない**まま作業開始
   - （または、他の誰かが最近PRをマージしたと仮定）
2. 新しいブランチ `feature/old-start-あなたの名前` を作成
3. 何かしら変更してコミット→プッシュ→PR作成
4. GitHub上のPR画面で「**This branch is out-of-date with the base branch**」と表示される

**対処法（3パターン）**：

**パターンA：GitHub上の「Update branch」ボタン（一番簡単）**
- PR画面に「Update branch」ボタンが表示される
- クリックするだけで自動的にmainの最新を取り込む
- **初心者はまずこれでOK**

**パターンB：ローカルでpullして最新化**
1. GitHub Desktopで「Current branch」をmainに切り替え
2. 「Fetch origin」→「Pull origin」で最新を取得
3. 自分のブランチに戻る
4. 「Branch」メニュー →「Update from main」を選択

**パターンC：コンフリクトまで発生した場合**
- 課題3と同じ手順でコンフリクトを解決

**予防策（一番大事）**：

毎日の作業開始前に以下を習慣にすると、このトラブルはほぼ防げます：

```
① GitHub Desktop を開く
② 「Current branch」を main に切り替え
③ 「Fetch origin」ボタンをクリック
④ 「Pull origin」ボタンが出たら必ず実行
⑤ それから新しいブランチを切って作業開始
```

**ゴール**：
- 「pullを忘れた」トラブルを体験しておく（本番で慌てない）
- 作業開始前のpull習慣を身につける
- 「Update branch」ボタンの位置を知っておく

### 課題6：間違えた時の取り消し方

「やらかした」を安全に戻す練習です。**ミスしても復帰できる**という安心感を得ることが目的です。

#### パターンA：保存しただけでコミット前の場合

1. `index.html` をわざと適当に編集してぐちゃぐちゃにする（例：文字を全部消す）
2. エディタで**保存**する
3. GitHub Desktopを開く →「**Changes**」タブに変更が表示される
4. ファイル名を**右クリック** →「**Discard changes**」をクリック
5. 確認ダイアログで「Discard changes」をクリック
6. 変更が取り消され、編集前の状態に戻る

→ **この操作は元に戻せません**。本当に破棄していい時のみ使用してください。

#### パターンB：コミットまでしてしまった場合（Revert）

1. 練習用のブランチで何かしらの変更をしてコミット
2. GitHub Desktopの「**History**」タブを開く
3. 取り消したいコミットを**右クリック** →「**Revert this commit**」を選択
4. 「Revert "コミット名"」という新しいコミットが自動で追加される
5. これでコミット前の状態に戻る（コミット履歴は残る）

→ 履歴を消すのではなく「打ち消すコミット」を積む方式なので**安全**です。

#### パターンC：間違えてmainブランチで直接編集してしまった

これが一番焦るパターン。落ち着いて対応しましょう。

**変更を取り消したい場合**：
- パターンAと同じ手順で「Discard changes」

**変更は残したい（ブランチに切り出したい）場合**：
1. 変更は保存した状態のまま GitHub Desktop を開く
2. 「**Current branch**」→「**New branch**」で新しいブランチを作成
   - ブランチ名：`feature/救出-あなたの名前`
   - **この時、現在の変更は自動的に新ブランチに持ち越されます**
3. そのまま新ブランチでコミット・プッシュ・PR作成
4. mainは変更されていない状態に戻る

→ **変更が消えないので安心**。焦らなくて大丈夫です。

#### パターンD：間違えてpushしてしまった場合

すでにGitHubに反映してしまった変更を取り消したい時。

1. GitHub上でPRを作る前の状態なら、パターンB（Revert）で新しい「取り消しコミット」をpushする
2. PRを出してしまっている場合は、PRを「Close」すれば未マージの変更は反映されない
3. マージまでしてしまっている場合は、責任者に相談。Revertで戻すか、新しいPRで修正する

**ゴール**：
- 「ミスしても戻せる」という安心感を得る
- パターンA〜Dの使い分けができる
- mainブランチ直接編集の対処法を知る（絶対やる人が出るので予防教育）

### 課題7：色を変えてみる（応用編＆レビュー却下の練習）

`css/style.css` の色を変えて、サイト全体の雰囲気を変えてみます。

1. 新しいブランチ `feature/color-change-あなたの名前` を作成
2. `css/style.css` の先頭（`:root` ブロック）を確認：
   ```css
   :root {
     --green-dark: #2d5f3f;
     --green: #4a8b5d;
     ...
   }
   ```
3. 好きな色コードに変更（[https://colorhunt.co](https://colorhunt.co) で色を探す）
4. コミット → プッシュ → PR作成

**この課題はレビューで却下してOK**（**Request changes** の練習として使いましょう）。
責任者は「会社のブランドカラーから外れるのでこの色はNG」など理由を付けて、Request changesを出す練習ができます。

**ゴール**：
- CSS変更でも同じgitフローが使えることを確認
- 責任者が **Request changes** を実際に出す経験を積む
- 「却下されても作業は無駄にならない」「再提出すればいい」ことを体感

## ❓ よくあるトラブル

### Q. 「Push origin」してもGitHubに反映されない
→ GitHub Desktopの右上が「Fetch origin」になっているか確認。`Push origin`ボタンがあればクリック。

### Q. コンフリクトが起きて怖い
→ 慌てずに責任者に相談してください。コンフリクトは正常な現象です。

### Q. 間違えてmainブランチで直接編集してしまった
→ 課題6のパターンCを参照。**変更は消えないので焦らなくてOK**。

### Q. プルリクエストって何？
→ 「自分のブランチの変更を main に取り込んでほしい」という申請のこと。責任者がレビューして問題なければ承認＆マージします。

### Q. Revertしたら履歴が消える？
→ **消えません**。Revertは「打ち消すコミット」を新たに積む方式なので、履歴は全て残ります。
